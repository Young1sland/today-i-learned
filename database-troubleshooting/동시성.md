# 동시성 이슈

## 1. 동시 요청에 따른 데이터 중복 생성 문제

```text
문제

같은 사용자가 같은 쿠폰 발급이 동시에 여러번 들어오는 경우 어떻게 막을 것인가?
```

### 잘못된 사례: 미리 조회(select)하고 발급 받았으면 막는다 (X)

```text
동시에 두 요청이 들어오면:

요청 A: 조회 → 없음
요청 B: 조회 → 없음
요청 A: insert 성공
요청 B: insert 성공
=> 막지 못함. 동시에 발급될 수 있다.
```

### 모범 사례

중복 방지는 애플리케이션 로직에만 의지하면 안됨.
DB의 unique constraint를 최종 보장 장치로 둬야 함.

```sql
// db unique index로 중복 방지
UNIQUE KEY uq_user_coupon (user_id, coupon_id)
```

- unique 걸어두고 insert 시 duplicate key error가 나면 기발급 쿠폰 에러 처리
- 데이터 정합성의 최종 보장은 DB에서 한다.

## 2. Idempotency-Key(아이덤포턴시 키)

실제 결제 API에 많이 쓰는 방식으로 네트워크 retry 문제, 클라이언트의 중복 요청을 방지.

보통은 동일한 키의 요청이 다시 들어오면 첫번째 요청의 결과를 그대로 반환하도록 설계

처리 흐름
1️⃣ 키 없음 → processing 저장
2️⃣ 비즈니스 로직 수행
3️⃣ 결과 저장
4️⃣ 다음 요청은 같은 결과 반환

```text
예: 결제 API 호출 시

Header
Idempotency-Key: 8d83e9e4

- 처음 요청
  Redis에 저장. NX: 키가 없을 때만 생성
    SET idem:8d83e9e4 processing NX EX 300

- 결제 완료 후
    SET idem:abc123 '{"paymentId":123,"status":"success"}' EX 86400

- 다음 중복 요청 시에는 저장된 값 리턴 
```

```ts
const key = `idem:${idempotencyKey}`

// 1️⃣ 먼저 조회
const cached = await redis.get(key)

if (cached && cached !== "processing") {
  return JSON.parse(cached) //결제 완료 후에는 같은 결과 리턴함
}

// 2️⃣ lock 시도
const lock = await redis.set(key, "processing", "NX", "EX", 30)

//key 생성 성공하면 "OK" 리턴
//이미 키가 존재하는 경우 null 리턴
if (!lock) {
  throw new Error("duplicate request") //이미 키가 존재
}

// 3️⃣ 결제 처리
const result = await processPayment()

// 4️⃣ 결과 저장
await redis.set(key, JSON.stringify(result), "EX", 86400)

return result
```

## 3. Transaction(트랜잭션)

트랜잭션을 사용하면 여러 DB 작업을 하나의 원자적 작업으로 묶을 수 있어서
일부만 반영되는 상황을 방지하고 데이터 정합성을 유지 가능하다.

정합성을 트랙잭션으로 보장. 여러 DB 작업이 전부 성공하거나 전부 실패하게 만든다.

```text

예제
  계좌 이체
  A → B 1000원
 1️⃣ A 잔액 감소
 2️⃣ B 잔액 증가

SQL
    UPDATE account
    SET balance = balance - 1000
    WHERE id = 'A';

    UPDATE account
    SET balance = balance + 1000
    WHERE id = 'B';

문제 상황
    1번 성공했으나 2번 실패
    A 계좌 -1000, B 계좌 +0
    => 1000원 사라짐. 정합성 깨진 상태가 됨

실패할 경우 모두 Rollback이 필요하다.
트랜잭션 사용
    BEGIN;

    UPDATE account
    SET balance = balance - 1000
    WHERE id = 'A';

    UPDATE account
    SET balance = balance + 1000
    WHERE id = 'B';

    COMMIT;
COMMIT 전 실패하면
    ROLLBACK, 모두 취소
```

```ts
await dataSource.transaction(async (manager) => {
  await manager.update(Account, { id: 'A' }, {
    balance: () => "balance - 1000"
  })

  await manager.update(Account, { id: 'B' }, {
    balance: () => "balance + 1000"
  })

})
```

## 4. 재고 차감 문제

```text
문제.

재고가 1개 남은 상품을 100명이 동시에 주문하면 어떻게 처리할 것인가?
```

### 트랜잭션을 사용해도 race condition은 생길 수 있다

```text
문제 상황 (재고 차감)

- 재고 조회 후 0보다 크면 stock-1로 update
    BEGIN;

    SELECT stock FROM product WHERE id = 1;

    -- 애플리케이션에서
    if (stock > 0) {
        UPDATE product
        SET stock = stock - 1
        WHERE id = 1;
    }

    COMMIT;

- A,B 두 요청이 동시에 들어올 때 실행 흐름
    A
    BEGIN
    SELECT stock = 1
    B
    BEGIN
    SELECT stock = 1

    둘 다 재고 1을 보고 차감 가능한 걸로 처리

    A
    UPDATE stock = 0
    COMMIT
    B
    UPDATE stock = -1
    COMMIT

    결과
    재고 1개인데 2명이 구매 성공

```

### 해결 방법 1. 조건부 UPDATE

실무에서 가장 많이 쓰는 방식.

- 쿼리 한 번. 원자적 처리
- 락을 길게 잡지 않음
- 구현 단순. 성능도 좋음

참고로 UPDATE query이므로 결국 내부적으로는 row lock이 걸리지만 lock 유지시간이 짧다.

```sql
UPDATE product
SET stock = stock - 1
WHERE id = 1
AND stock > 0; //stock이 0인 경우만 UPDATE

결과 
affected rows = 1 → 성공
affected rows = 0 → 실패

```

### 해결 방법 2. row lock

Select 시 FOR UPDATE query 추가하여 row lock을 잡는다.(비관적 Lock)

```text
BEGIN;

SELECT stock
FROM product
WHERE id = 1
FOR UPDATE;

-- 재고 확인
-- 차감

UPDATE product
SET stock = stock - 1
WHERE id = 1;

COMMIT;


첫 번째 트랜잭션이 row lock을 잡고 두 번째 트랜잭션은 대기한다.
```

### 비관적 락

충돌이 자주 난다고 가정하고 먼저 데이터에 락을 거는 방법으로 보통 `FOR UPDATE`로 row lock을 걸어 구현한다.

- 장점
  - 충돌이 심한 상황에서 안전
  - 복잡한 비즈니스 검증에 유리
- 단점
  - 대기 증가. throughput 단위 시간당 처리량 떨어짐
  - 데드락 가능성
  - 처리량 저하 가능

복잡한 비지니스 검증이 필요한 경우가 아니라면 먼저 더 단순한 원자적 처리 방식(조건부 update 등)을 검토하는 것이 좋다.

비관적 락을 쓰면 좋은 경우

1. 조회 후 복잡한 비즈니스 판단이 필요한 경우
   재고,쿠폰 상태, 사용자 등급, 기간 조건, 테이블 상태 보고 처리해야 할 때
2. 충돌 가능성이 매우 높고 재시도 비용이 큰 경우
3. 반드시 순차적으로 처리해야 하는 경우

### 낙관적 락

충돌이 거의 안난다고 가정하고, 미리 락을 잡지 않는다.
보통 version 컬럼을 통해 데이터 바뀌었는지 확인하고 충돌이 나면 실패 처리한다.

- 장점
  - Lock을 잡지 않음 → 다른 트랜잭션 대기 없음
  - 성능 좋음 → read-heavy 시스템에 좋음
  - 데드락 없음

게시글 프로필, 상품 설명 수정 등에 쓰임
충돌 나면 실패로 처리하고 사용자에게 재시도하도록 유도

```sql
예시. 게시글 수정

테이블
    post
    -------
    id
    title
    content
    version

초기 데이터
    id=1
    title="hello"
    version=5

사용자 A 
    게시글 조회
        title="hello"
        version=5

    수정 후 요청
        UPDATE post
        SET title="hello world",
            version = version + 1
        WHERE id = 1
        AND version = 5;
        → 성공. version=6
사용자 B
    처음 조회했을 때 version=5
    동시에 수정 시도
        UPDATE post
        SET title="new title",
            version = version + 1
        WHERE id = 1
        AND version = 5;
        → version=6이므로 실패. affected rows = 0


서버는 보통 409 Conflict 에러 리턴
자동 재시도X. 게시글이 다른 사용자에 의해 수정되었습니다.
```

```ts
//코드 예
const result = await repo.update(
  { id: postId, version: oldVersion },
  {
    title: newTitle,
    version: () => "version + 1"
  }
)

if (result.affected === 0) {
  throw new ConflictException("post updated by another user")
}
```

충돌 가능성이 낮고 읽기가 많은 경우에는 낙관적 락이 효율적이고,
충돌이 자주 발생하거나 순차 처리가 필요한 경우에는 비관적 락을 고려합니다.

#### TypeOrm VersionColumn

@VersionColumn()은 TypeORM에서 낙관적 락(optimistic locking)을 위한 버전 컬럼

주의: save() 시에만 동작한다. update(), increment(),decrement() 동작X

```ts
@VersionColumn()
version: number
```

```ts
try {
    const post = await repo.findOneBy({ id: 1 })
    post.title = "new title"
    await repo.save(post)
} catch (e) {
  if (e instanceof OptimisticLockVersionMismatchError) {
    throw new ConflictException("post updated by someone else")
  }
}

이렇게 동작
UPDATE post
SET title = 'new title',
    version = version + 1
WHERE id = 1
AND version = 3

```

## Redis의 적절한 용도

메모리 기반으로 속도가 빠르고 캐시, 세션, rate limiting, 분산락 등의 일시적 상태 관리에 유리

1. 과도한 DB 부하 완화
2. 대기열 / 진입 제어: 한 번에 너무 많은 요청이 DB로 가는 걸 줄임.
3. 캐시: 조회성 데이터 가속.
