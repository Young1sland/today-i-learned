## Cache Behavior
CloudFront의 요청이 어떻게 처리되는지 정의하는 다양한 설정의 모음
- CloudFront의 요청이 어떻게 처리되는지 정의하는 다양한 설정의 모음
- 경로 패턴 단위로 Behavior를 구성
  - 예: files/*, files/*.png, *.png
  - 경로 패턴 목록 중 처음으로 매칭된 패턴의 동작을 적용
  - 신규 생성 시 *로 고정
- 주요 구성 내용
  - Origin
  - 뷰어 설정
  - Cache Policy, Viewer Response Policy, Lambda@EDGE 연결
### Viewer 설정
- Viewer 프로토콜
  - CloudFront에 접근하는 프로토콜
  - HTTP and HTTPS
  - Redirect HTTP to HTTPS
  - HTTPS Only (HTTP로 요청하면 에러남)
- HTTP Method
  - GET, HEAD, OPTIONS. PUT, POST, PATCH, DELETE
- 뷰어 액세스를 제한: Presigned URL/Presigned Cookie로만 접근 가능하도록 할지 여부. 특정 컨텐츠에 대해 승인된 사용자에게만 접근 허용시킴.
### Policy 설정
- Cache Policy
   - 어떤 키(HTTP Header, 쿠키, querystring 등)으로 컨텐츠를 캐시하는지
   - 얼마나 오래 캐시하는지(TTL)
   - 컨텐츠를 압축 저장 관련 설정
- Origin Request Policy
  - Origin에 컨텐츠를 요청할 때 어떤 내용을 전달할 것인지(HTTP Header, querystring 등)
  - Cache Key로 사용할지 여부와 상관없이 독립적으로 설정 가능
- Response Headers Policy
  - 응답 과정에서 어떤 HTTP Header를 제거하거나 더할지 설정
  - 예를 들면 S3의 metadata가 모두 전달이 되는데 민감한 데이터 삭제가 필요한 경우
  - 최대 10개까지 가능. Authorization header만 불가능
- 예시1
  1. Viewer는 /img.png?color=red&user_id=chris에 접근
  2. Cache Policy는 url path, querystring color를 사용하고 있음. 이 경우 user_id는 cache key가 아니므로 origin에 전달X
  3. 그러나 origin에서는 cache key가 아니지만 user_id를 받아 로깅에 사용하고 싶다. 이 경우 Origin Request Policy에 user_id를 설정하여 받을 수 있다.
- 예시2
  1. Viewer는 cloudfront를 통해 s3에 접근한다. img.png파일을 받아오는데 이 파일에는 user_id,phone과 같은 metadata가 포함되어있다.
  2. phone과 같은 민감한 정보를 제거하고 service:prodapp이라는 header를 더해 response 보내고 싶다. 이경우 Response Header Policy에 설정할 수 있다.
