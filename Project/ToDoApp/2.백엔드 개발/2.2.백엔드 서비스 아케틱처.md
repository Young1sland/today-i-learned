### DTO
서비스가 요청을 처리하고 DTO(Data Transfer Object)로 변환하여 클라이언트로 리턴한다. 사용 이유는 하기와 같다.
- 비지니스 로직 캡슐화. 외부 사용자에게 서비스 내부 로직,데이터베이스 구조 숨김
- 클라이언트가 필요한 정보를 모델이 전부 포함하지 않는 경우가 많음. ex) Error message

### REST API
REST(Representational State Transfer) 아키텍처 스타일 제약 조건
- 클라이언트-서버     
  리소스 관리하는 서버 존재. 다수의 클라이언트가 네트워크 통해 서버에 접근
- Stateless(상태 없음)    
  클라이어느가 서버에 요청을 보낼 때 이전 요청의 영향을 받지 않음.
- 캐시 가능한 데이터(Cacheable)      
  서버에서 리소스 리턴할 때 캐시 가능한지 명시 가능해야 함.      
  HTTP에서는 cache-control이라는 헤더에 리소스의 캐시 여부 명시할 수 있음.
- 일관적인 인터페이스(Uniform Interface)      
  URI,요청,응답의 형태가 일관성이 있어야 함.
- 레이어 시스템(Layered System)      
  클라이언트가 서버에 요청을 보낼 때 여러 개의 레이어로 된 서버를 거칠 수 있음.
- 코드 온-디멘드(Code-On-Demand) - 선택사항      
  클라이언트는 서버에 코드를 요청할 수 있고 서버가 리턴한 코드를 실행할 수 있다.

### RestController
@RestController는 크게 @Controller, @ResponseBody 두 어노테이션으로 이루어져 있다.
- @ResponseBody : 메서드가 리턴할 때 스프링은 리턴된 오브젝트를 JSON 형태로 바꾸고 HttpResponse에 담아 반환한다는 뜻이다. (Serialization : Obj -> JSON)

### Class를 Entity로 정의 시 주의점
- NoArgsConstructor 필요
- Getter/Setter 필요
- Primary Key 지정 필요

### 스프링이 JpaRepository를 실행하는 방법?
- 스프링은 MethodInterceptor라는 AOP 인터페이스를 이용한다.
- MethodInterceptor는 JpaRepository 메서드를 부를 때마다 메서드 콜을 가로채 간다. 가로챈 메서드의 이름을 확인하고 기반으로 쿼리를 작성한다.

### Query 메서드 작성
https://docs.spring.io/spring-data/jpa/docs/current/reference/html/#jpa.query-methods.query-creation
```java
@Repository
public interface TodoRepository extends JpaRepository<TodoEntity, String> {
    List<TodoEntity> findByUserId(String userId);

    //복잡한 쿼리는 @Query 사용. ?1은 매개변수의 순서 위치
    @Query("select * from TodoEntity t where t.userId = ?1")
    List<TodoEntity> findByUserIdQuery(String userId);
}
```
