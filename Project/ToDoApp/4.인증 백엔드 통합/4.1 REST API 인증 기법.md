# REST API 인증 기법

## Basic 인증
- HTTP 요청 헤더의 Authrization 부분에 아이디와 비밀번호를 콜론으로 붙인 후 Base64 인코딩한 문자열 보냄. 디코더만 있으면 디코딩할 수 있으므로 반드시 HTTPS와 사용해야 함
    ```Authorization: Basic XXXXXXX==```
- 인증 서버와 인증 DB에 과부하 걸릴 확률 높음. 
- 인증 서버가 단일 장애점(Single Point of Failure)이 될 수 있음.

## 토큰 기반 인증
- 로그인 시 서버가 토큰을 만들어 반환. 클라이언트는 이후 요청에 토큰을 넘겨 인증
    ```Authorization: Bearer XXXXX```
- 서버가 임의로 토큰 생성. 유효기간 정할 수 있음
- 세션 기반 인증과 거의 같으며 스케일 문제를 해결할 수 없음

## JWT(JSON WEB TOKEN)
- 서버에 의해 전자 서명(Digital Signature)된 토큰을 이용해 스케일 문제를 해결할 수 있다.
- JWT : 전자서명된 토큰 중 하나로 JSON 형태로 된 TOKEN
- 전자서명 : 서명하고 싶은 메시지를 해시함수를 이용해 축약 후 개인키로 암호화했을 때 나오는 값
```
{header}.{payload}.{signature}
{    //HEADER
    "typ:"JWT", //Type. Token Type
    "alg":"HS512" //해시 알고리즘 종류
}.
{   //payload
    "sub": "402XX" //Subject, Token 주인. ID처럼 유일한 식별자여야 함.
    "iss":"demo app", //Issuer. 토큰 발행 주체. 페이스북이 발행하면 facebook
    "iat":1595733657, //issued at, 발행 날짜 시간
    "exp":1596597657  //expiration, 만료 시간
}.
Nn4d1MOVXXX.... //Signature, 토큰 발행한 주체 Issuer가 발행한 서명. 토큰 유효성 검사에 사용
```
- 로그인 시 사용자의 ID, PW으로 인증 후 정보를 이용해 {Header},{payload} 작성
- Secret key로 {header}.{payload}부분을 전자서명하고 {header}.{payload}.{signature}로 이어 붙인 뒤 Base64로 인코딩 한 후 반환
- 토큰으로 리소스 접근 요청 시 서버는 Base64로 디코딩. 얻은 JSON의 {header}.{payload}와 Secet key로 전자 서명 만들어 {signature} 부분과 비교해 유효성 검사 함.
- 토큰의 유효성 검사 시 인증 서버에 물어볼 필요가 없음. 인증 서버 부하 줄이고 단일 장애점이 아니게 함. (근데 유효성 검증 후 사용자 정보를 가져온다면 똑같은 거 아님?..)
