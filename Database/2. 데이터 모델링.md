### 데이터 모델링의 특징
- 추상화(Abstraction) : 현실 세계를 간략하게 표현
- 단순화(Simplification) : 쉽게 이해할 수 있도록 표현
- 명확성(Clarity) : 명확하게 의미가 해석되어야 하고 한가지 의미를 가짐

### 데이터 모델링 단계
1. 개념적 모델링(Conceptual Data Modeling)    
   엔티티와 속성을 도출하고 개념적 ERD(Entity Relationship Diagram) 작성. 전사적 관점,업무 측면에서 모델링
2. 논리적 모델링(Logical Data Modeling)   
   식별자를 도출하고 필요한 모든 릴레이션을 정의. 정규화 수행하여 데이터 모델의 독립성 확보. 특정 데이터베이스 모델에 종속.
3. 물리적 모델링(Physical Modeling)    
   데이터베이스 실제 구축. 테이블,인덱스,함수 등을 생성. 성능,보안,가용성 고려.

### 데이터 모델링 관점
- 데이터 : 구조,정적 분석
- 프로세스 : 수행하는 작업. 시나리오 분석,도메인,동적 분석
- 데이터와 프로세스 : 데이터와 프로세스 간의 관계, CRUD 분석

### ERD(Entity Relationship Diagram) 절차
1. 엔티티 도출
2. 엔티티 배치   
   중요한 엔티티는 왼쪽 상단에 배치
3. 엔티티 관계 설정
4. 관계명 서술   
   엔티티 간의 행위나 존재 표현
5. 관계 참여도 표현   
   한 개의 엔티티와 다른 엔티티 간의 참여하는 관계 수 의미(1대1, 1대다..)
6. 관계의 필수 여부 표현

### 데이터 모델링 고려사항
1. 독립성   
   중복된 데이터 제거. 정규화
2. 고객 요구사항 표현   
   요구사항을 간결하고 명확하게 표현.
3. 데이터 품질 확보   
   데이터 표준 정의하고 표준 준수율 관리


### 3층 스키마(3-Level Schema)
- 사용자, 설계자, 개발자의 관점에 따라 데이터베이스를 기술하고 관계를 정의한 ANSI 표준
- 독립성 확보 위한 방법으로 각 계층을 뷰라고도 함
- 독립성
  - 논리적 독립성 : 개념 스키마가 변경되어도 외부 스키마가 영향 받지 않음.
  - 물리적 독립성 : 내부 스키마가 변경되어도 개념 스키마가 영향 받지 않음.   
- 구조
  - 외부 스키마(External Schema)   
    사용자 관점. 업무상 데이터 접근. 응용프로그램이 접근하는 데이터베이스 정의
  - 개념 스키마(Conceptual Schema)   
    설계자 관점. 전체 데이터베이스 내의 규칙과 구조 표현. 통합 데이터베이스 구조
  - 내부 스키마(Internal Schema)   
    개발자 관점, 데이터베이스의 물리적 저장 구조. 레코드 구조,필드 정의,인덱스 등
    >ANSI SQL 표준 : 미국표준협회 SQL 표준은 SQL문이 모든 DBMS에서 호환되도록 정의한 표준

### 엔티티
- 업무에서 관리해야 하는 데이터 집합을 의미
- 특징
  - 유일한 식별자가 있어야 한다.
  - 2개 이상의 인스턴스가 있어야 한다.
  - 반드시 속성을 가진다.
  - 다른 엔티티와 최소한 한 개 이상 관계가 있어야 한다.
  - 업무에서 관리되어야 하는 집합이다.

### 릴레이션,테이블, 인스턴스
- 릴레이션과 테이블은 같은 의미로 보면 됨. 릴레이션에 기본키,제약조건 설정하면 테이블이 된다.
- Relationship은 릴레이션 간의 관계
- 인스턴스는 릴레이션이 가질 수 있는 값을 의미

### 엔티티 종류
- 유무형에 따른 종류
  - 유형 엔티티 : 업무에서 도출되며 지속적으로 사용되는 엔티티 (고객,사원 등)
  - 무형 엔티티 : 물리적 형태 없고 개념적으로 사용되는 엔티티(거래소 종목, 생명보험 상품 등)
  - 사건 엔티티 : 비지니스 프로세스를 실행하면서 생성되는 엔티티 (주문, 취소 등)
- 발생 시점에 따른 종류
  - 기본 엔티티(Basic Entity) : 키 엔티티. 독립적으로 생성되는 엔티티(고객 등)
  - 중심 엔티티(Main Entity) : 기본 엔티티로부터 발생되고 행위 엔티티를 생성하는 것(계좌,주문,취소 등)
  - 행위 엔티티(Active Entity) : 2개 이상의 엔티티로부터 발생 됨.(주문 이력 등)
  
### 속성
- 엔티티가 가지는 항목. 더 이상 분리되지 않는 단위. 주식별자에게 함수적으로 종속 됨.
- 종류
  - 분해 여부에 다른 종류
    - 단일 속성
    - 복합 속성 : 여러 개의 의미가 있는 것(주소, 시군동으로 분해될 수 있음)
    - 다중값 속성 : 여러 개의 값을 가질 수 있는 것. 엔티티로 분해된다.(상품 리스트,..)
  - 특성에 따른 종류
    - 기본 속성 : 본래의 속성 (회원 ID,이름,..)
    - 설계 속성 : 데이터 모델링 과정에서 발생되는 속성. 유일한 값 부여(상품코드, 지점 코드 등)
    - 파생 속성 : 다른 속성에 의해 만들어지는 속성 (합계,평균)
    >도메인 : 속성이 가질 수 있는 값의 범위

### 관계(Relationship)
엔티티 간의 관련성을 의미하며 존재관계와 행위관계로 분류 됨
- 종류
  - 존재 관계 : 엔티티 간의 상태 나타냄. 엔티티 간의 존재 여부의 관계. 
  - 행위 관계 : 엔티티 간에 어떤 행위에 있는 것
- 관계 차수
  - 두 개의 엔티티 간에 관계에 참여하는 수를 의미 (1:1, 1:N,..)
  - 종류
    - 1:1 : 반드시 존재하면 완전1대1, 없을수도 있으면 선택적1대1
    - 1:N
    - M:N : M대N 관계의 join은 카테시안 곱이 발생. 1대N,N대1로 해소 필요
    - 필수적 관계는 "|"로 표현. 선택적 관계는 "O"로 표현
- 식별 관계와 비식별 관계
  - 식별 관계(Identification Relationship)
    - 독립적으로 존재할 수 있는 Strong Entity가 다른 엔티티의 기본키의 하나로 자신의 기본키를 공유하는 관계
    - 예 : 고객 엔티티의 기본키 ID가 계좌 엔티티의 기본키의 하나로 공유되는 경우
  - 비식별 관계(Non-Identification Relationship)
    - Strong Entity의 기본키가 다른 엔티티의 기본키가 아닌 일반 칼럼으로 관계 가지는 것
    - 예 : 부서 엔티티의 code가 사원 엔티티의 일반 컬럼에 포함되는 경우

### 엔티티 식별자(Entity Identifier)
엔티티를 대표할 수 있는 유일성을 만족하는 속성
- 종류
  - 대표성 여부에 따른 종류
    - 주식별자 : 유일성,최소성 만족하면서 대표성을 가지는 식별자. 다른 엔티티와 참조 관계 연결  가능
    - 보조 식별자 : 유일성,취소성 만족하지만 대표성 만족 못하는 식별자
  - 생성 여부에 다른 종류
    - 내부 식별자 : 엔티티 내부에서 스스로 생성되는 식별자(주문번호 등)
    - 외부 식별자 : 다른 엔티티와의 관계로 인해 만들어지는 식별자(예 : 계좌의 회원ID)
  - 속성 수에 따른 종류
    - 단일 식별자 : 하나의 속성으로 구성
    - 복합 식별자 : 두 개 이상의 속성으로 구성
  - 대체 여부에 따른 종류
    - 본질 식별자 : 비지니스 프로세스에 의해 만들어지는 식별자
    - 인조 식별자 : 인위적으로 만들어지는 식별자   
                 주식별자로 설정할 것이 없거나 많을 경우 임의로 생성. (예 : Sequence Number,..) 
                 
- 데이터 베이스 키의 종류
  - 기본키(Primary Key) : 후보키 중에서 엔티티를 대표할 수 있는 키
  - 후보키(Candidate Key) : 유일성과 최소성을 만족하는 키
  - 슈퍼키(Super key) : 유일성은 만족하지만 최소성 만족하지 않는 키
  - 대체키(Alternate Key) : 후보키 중 기본키 선정하고 남은 키
  - 외래키(Foreign key) : 다른 테이블의 기본 키를 가리키는 키. 참조 무결성 확인 위해 사용
